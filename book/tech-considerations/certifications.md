# การรับรองและทดสอบความสอดคล้องของเทคโนโลยี PQC

การนำเทคโนโลยีรหัสลับหลังควอนตัม (PQC) ไปใช้งานในระบบสารสนเทศขององค์กร ไม่ได้สิ้นสุดเพียงแค่การเลือกอัลกอริทึมที่ถูกต้องหรือการเลือกไลบรารีซอฟต์แวร์ที่เหมาะสม สิ่งที่สำคัญอย่างยิ่งสำหรับองค์กรที่ต้องปฏิบัติตามกฎระเบียบ โดยเฉพาะในภาคการเงิน ภาครัฐ และโครงสร้างพื้นฐานสำคัญ คือการตรวจสอบให้แน่ใจว่าผลิตภัณฑ์และโมดูลรหัสลับที่ใช้งานนั้น **ได้รับการรับรองอย่างเป็นทางการ** (Formally Certified) และ **ผ่านการทดสอบความสอดคล้อง** (Conformance Testing) กับมาตรฐานที่เกี่ยวข้อง

ในยุคของการเปลี่ยนผ่านสู่ PQC กระบวนการรับรองกำลังอยู่ในช่วงเปลี่ยนผ่านเช่นกัน หน่วยงานออกใบรับรองต้องปรับตัวรองรับอัลกอริทึมใหม่ที่มีโครงสร้างทางคณิตศาสตร์แตกต่างจากเดิมอย่างมาก การทำความเข้าใจภูมิทัศน์ของการรับรองในปัจจุบันจึงมีความสำคัญอย่างยิ่งต่อการวางแผนการจัดซื้อและการปฏิบัติตามกฎหมายขององค์กรไทย

## ทำไมการรับรองและการทดสอบจึงสำคัญ

การนำ PQC ไปใช้งานอย่างไม่ถูกต้อง (Incorrect Implementation) เป็นภัยคุกคามที่สำคัญไม่แพ้ภัยจากคอมพิวเตอร์ควอนตัม อัลกอริทึมที่ถูกต้องทางทฤษฎีอาจมีช่องโหว่ร้ายแรงเมื่อนำไปพัฒนาในซอฟต์แวร์ที่ไม่ระมัดระวัง เช่น:

- **การรั่วไหลของข้อมูลผ่านช่องทางข้างเคียง (Side-Channel Leakage):** การคำนวณที่ไม่ใช่ Constant-time ทำให้ผู้โจมตีวิเคราะห์ค่าความล่าช้าของการประมวลผล (Timing Attack) เพื่อสืบค้นกุญแจลับได้
- **การสร้างตัวเลขสุ่มที่อ่อนแอ (Weak Random Number Generation):** ML-KEM และ ML-DSA ต้องพึ่งพาการสุ่มที่มีคุณภาพสูงอย่างเคร่งครัด ความอ่อนแอในตัวสร้างตัวเลขสุ่ม (RNG) นำไปสู่ช่องโหว่ที่ทำให้ระบบถูกทำลายได้ทั้งหมด
- **การปรับแต่งพารามิเตอร์ที่ผิดพลาด (Incorrect Parameter Configuration):** การเลือกพารามิเตอร์ที่ไม่ตรงตามมาตรฐาน NIST อาจทำให้ระดับความปลอดภัยต่ำกว่าที่คาดไว้

กระบวนการรับรองและทดสอบความสอดคล้องออกแบบมาเพื่อตรวจจับและกำจัดข้อบกพร่องเหล่านี้อย่างเป็นระบบ ก่อนที่ผลิตภัณฑ์จะถูกนำไปใช้งานในสภาพแวดล้อมจริง

## มาตรฐานการรับรองหลักที่เกี่ยวข้องกับ PQC

### FIPS 140-3 และโครงการ CMVP

**FIPS 140-3** (Federal Information Processing Standard 140-3) คือมาตรฐานความปลอดภัยสำหรับโมดูลรหัสลับ (Cryptographic Module) ที่ออกโดย NIST ซึ่งเป็นมาตรฐานอ้างอิงหลักสำหรับหน่วยงานรัฐบาลสหรัฐอเมริกาและองค์กรที่ต้องการสาธิตความปลอดภัยระดับสากล มาตรฐานนี้กำหนด 4 ระดับความปลอดภัย (Security Level 1-4) ตามความเข้มข้นของการป้องกันทั้งด้านซอฟต์แวร์และฮาร์ดแวร์

**โครงการ CMVP** (Cryptographic Module Validation Program) คือกลไกที่ NIST ร่วมกับ Canadian Centre for Cyber Security (CCCS) ใช้ในการรับรองโมดูลรหัสลับให้เป็นไปตาม FIPS 140-3 โดยผ่านห้องปฏิบัติการทดสอบอิสระที่ได้รับการรับรอง (Accredited Testing Laboratories หรือ ATLs)

**สถานะปัจจุบันสำหรับอัลกอริทึม PQC (ณ ต้นปี 2026):**

การรับรอง FIPS 140-3 สำหรับอัลกอริทึม PQC ยังอยู่ในช่วงเริ่มต้น เนื่องจาก NIST เพิ่งออกมาตรฐาน FIPS 203 (ML-KEM), FIPS 204 (ML-DSA) และ FIPS 205 (SLH-DSA) ในเดือนสิงหาคม 2024 ส่งผลให้:

- ผู้ผลิตโมดูลรหัสลับต้องยื่นขอการรับรองใหม่หรืออัปเดต Module Security Policy เพื่อรวมอัลกอริทึม PQC
- กระบวนการตรวจสอบโดย ATL ใช้เวลาโดยเฉลี่ย 12-24 เดือนสำหรับโมดูลใหม่
- ในระหว่างนี้ NIST อนุญาตให้ใช้กลยุทธ์ **Hybrid** ที่รวม PQC กับ ECC เพื่อให้ความปลอดภัยเพียงพอในช่วงเปลี่ยนผ่าน

**ผลกระทบต่อองค์กรไทย:**

องค์กรที่ต้องการผลิตภัณฑ์รหัสลับที่รับรอง FIPS 140-3 สำหรับ PQC โดยเฉพาะ จะต้องรอให้ผู้ผลิตได้รับการรับรองเสียก่อน ซึ่งอาจใช้เวลาอีก 1-3 ปี องค์กรควรตรวจสอบสถานะการรับรองจากฐานข้อมูล CMVP ที่ NIST ดูแลอยู่อย่างสม่ำเสมอ

### CAVP — โครงการตรวจสอบความถูกต้องของอัลกอริทึม

**CAVP** (Cryptographic Algorithm Validation Program) เป็นโครงการที่ NIST ใช้ในการทดสอบว่าการพัฒนาซอฟต์แวร์ (Implementation) ของอัลกอริทึมรหัสลับนั้นให้ผลลัพธ์ถูกต้องตรงตามมาตรฐาน โดยทดสอบผ่านชุดข้อมูลทดสอบมาตรฐาน (Known Answer Tests หรือ KATs) และ Test Vectors อย่างเป็นระบบ

**ความสำคัญของ CAVP สำหรับ PQC:**

NIST ได้เผยแพร่ Test Vectors อย่างเป็นทางการสำหรับ ML-KEM (FIPS 203), ML-DSA (FIPS 204) และ SLH-DSA (FIPS 205) ซึ่งนักพัฒนาและห้องปฏิบัติการทดสอบสามารถใช้เพื่อตรวจสอบว่าโค้ดที่พัฒนาขึ้นให้ผลลัพธ์เหมือนกันทุกประการกับมาตรฐาน การผ่าน CAVP เป็นขั้นตอนที่จำเป็นก่อนที่จะยื่นขอการรับรอง FIPS 140-3 ระดับโมดูล

**ประเภทของการทดสอบใน CAVP สำหรับ PQC:**

| ประเภทการทดสอบ | คำอธิบาย | อัลกอริทึมที่ใช้ |
| -- | -- | -- |
| **Known Answer Tests (KATs)** | ป้อนค่า Input คงที่แล้วตรวจสอบ Output ว่าตรงกับค่าอ้างอิง | ML-KEM, ML-DSA, SLH-DSA |
| **Monte Carlo Tests (MCTs)** | ทดสอบด้วยการวนซ้ำหลายพันรอบเพื่อตรวจสอบความสม่ำเสมอ | ใช้กับฟังก์ชัน Hash ที่ PQC อ้างอิง |
| **Component Tests** | ทดสอบองค์ประกอบย่อยอย่าง NTT, Keccak-f | ML-KEM (ใช้ SHAKE-128/256), ML-DSA |
| **RNG Validation** | ตรวจสอบคุณภาพของตัวสร้างตัวเลขสุ่มที่ใช้ร่วมกับ PQC | ทุกอัลกอริทึม |

### Common Criteria (CC) — การประเมินระดับผลิตภัณฑ์

**Common Criteria (CC)** หรือ ISO/IEC 15408 เป็นกรอบการประเมินความปลอดภัยของผลิตภัณฑ์ IT ระดับสากลที่ใช้กันอย่างแพร่หลายใน 31 ประเทศที่เป็นสมาชิก Common Criteria Recognition Arrangement (CCRA) รวมถึงสหรัฐอเมริกา สหภาพยุโรป ญี่ปุ่น และเกาหลีใต้

ต่างจาก FIPS 140-3 ที่มุ่งเน้นที่ตัวโมดูลรหัสลับ Common Criteria ประเมินความปลอดภัยของผลิตภัณฑ์โดยรวมในระดับ **Evaluation Assurance Level (EAL)** ตั้งแต่ EAL1 (ระดับต่ำสุด) ถึง EAL7 (ระดับสูงสุด)

**ความเกี่ยวข้องกับ PQC:**

สำหรับผลิตภัณฑ์ที่ต้องการการรับรอง Common Criteria เช่น Smart Cards, Hardware Security Modules หรือ Firewalls ที่รวม PQC เข้าไป การประเมิน CC จะต้องครอบคลุมหน้าที่รักษาความปลอดภัย (Security Functions) ใหม่ที่เกี่ยวข้องกับ PQC ผ่านการจัดทำ **Protection Profile (PP)** และ **Security Target (ST)** ที่อธิบายว่าผลิตภัณฑ์ปกป้องตัวเองจากภัยคุกคามอย่างไร

ปัจจุบัน กลุ่มทำงานด้าน Protection Profile ภายใต้ CCRA กำลังพัฒนา PP สำหรับผลิตภัณฑ์ที่รวม PQC โดยเฉพาะ คาดว่าจะเผยแพร่ PP ฉบับแรกในปี 2026-2027

### ETSI และการทดสอบความสอดคล้องสำหรับ PQC

**ETSI** (European Telecommunications Standards Institute) มีบทบาทสำคัญในการกำหนดมาตรฐานความสอดคล้องสำหรับ PQC ในบริบทของโทรคมนาคมและบริการดิจิทัลในยุโรป เอกสารมาตรฐานที่เกี่ยวข้องหลัก ได้แก่:

- **ETSI TS 103 744:** กำหนดข้อกำหนด Quantum-Safe KEM สำหรับบริการโทรคมนาคม รวมถึงพารามิเตอร์ความปลอดภัยขั้นต่ำ
- **ETSI TS 104 015:** ครอบคลุมการทดสอบความสอดคล้องสำหรับ ML-KEM ในบริบทของ TLS และโปรโตคอลเครือข่าย
- **ETSI TR 103 619:** คู่มือการย้ายระบบสู่ Quantum-Safe ที่รวมถึงข้อแนะนำการตรวจสอบความสอดคล้อง

สำหรับองค์กรไทยที่มีธุรกิจหรือความร่วมมือกับคู่ค้าในยุโรป หรือที่ใช้มาตรฐาน ETSI เป็นฐาน มาตรฐานเหล่านี้มีนัยสำคัญต่อการกำหนดข้อกำหนดสำหรับผู้ผลิตผลิตภัณฑ์

## กระบวนการทดสอบความสอดคล้อง (Conformance Testing Process)

### การทดสอบด้วย Test Vectors

Test Vectors คือชุดข้อมูลนำเข้า (Input) และผลลัพธ์ที่คาดไว้ (Expected Output) ที่สร้างขึ้นโดย NIST อย่างเป็นทางการ ซึ่งถือเป็นเครื่องมือพื้นฐานที่สุดในการตรวจสอบว่าการพัฒนา (Implementation) ของ PQC ให้ผลลัพธ์ที่ถูกต้อง

**แหล่ง Test Vectors สำหรับ PQC:**

- **NIST ACVP (Automated Cryptographic Validation Protocol):** ระบบทดสอบออนไลน์ของ NIST ที่ให้ผู้พัฒนาส่ง Implementation เพื่อทดสอบกับ Test Vectors ผ่าน API โดยอัตโนมัติ ปัจจุบันรองรับ ML-KEM, ML-DSA และ SLH-DSA แล้ว
- **โครงการ PQClean:** รวบรวม Reference Implementations พร้อม Test Vectors ที่ผ่านการตรวจสอบอย่างเข้มงวด
- **Repository ของ NIST บน GitHub:** เผยแพร่ Test Vectors อย่างเป็นทางการสำหรับ FIPS 203, 204 และ 205

**ตัวอย่างขั้นตอนการทดสอบด้วย Test Vectors สำหรับ ML-KEM:**

```
1. ดาวน์โหลด Test Vectors อย่างเป็นทางการจาก NIST ACVP
2. ป้อน Seed สำหรับ Key Generation ที่กำหนดใน Test Vector
3. ตรวจสอบว่า Public Key และ Private Key ที่ได้ตรงกับค่าอ้างอิง
4. ทดสอบฟังก์ชัน Encapsulation ด้วย Public Key ที่กำหนด
5. ตรวจสอบว่า Ciphertext และ Shared Secret ตรงกับค่าอ้างอิง
6. ทดสอบฟังก์ชัน Decapsulation และตรวจสอบ Shared Secret
7. ทดสอบ Decapsulation Failure (ป้อน Ciphertext ที่ผิดพลาด) เพื่อตรวจสอบการจัดการกรณีผิดพลาด
```

### การทดสอบช่องทางข้างเคียง (Side-Channel Testing)

การทดสอบ Side-Channel เป็นส่วนสำคัญที่แยกจากการทดสอบ Functional Correctness เนื่องจาก Implementation ที่ให้ผลลัพธ์ถูกต้องทางคณิตศาสตร์ยังอาจมีช่องโหว่เมื่อผู้โจมตีสามารถสังเกตลักษณะทางกายภาพของการทำงานได้

**ประเภทของ Side-Channel ที่เกี่ยวข้องกับ PQC:**

| ประเภทการโจมตี | สิ่งที่วัด | ความเสี่ยงสำหรับ PQC |
| -- | -- | -- |
| **Timing Attack** | ความแตกต่างของเวลาในการประมวลผล | ML-KEM Decapsulation ที่ไม่เป็น Constant-time |
| **Power Analysis (SPA/DPA)** | การใช้พลังงานขณะประมวลผล | NTT Operations ใน ML-KEM/ML-DSA |
| **Electromagnetic Analysis** | การแผ่สัญญาณแม่เหล็กไฟฟ้า | ชิป IoT ที่ไม่มีการป้องกัน |
| **Fault Injection** | การโจมตีด้วยการทำให้เกิดข้อผิดพลาด | การสร้างลายเซ็นที่บกพร่องทำให้หลุดกุญแจลับ |

ห้องปฏิบัติการที่มีความเชี่ยวชาญในการทดสอบ Side-Channel สำหรับ PQC ได้แก่ Riscure (เนเธอร์แลนด์), Rambus (สหรัฐอเมริกา) และ Fraunhofer AISEC (เยอรมนี)

### การทดสอบความสามารถในการทำงานร่วมกัน (Interoperability Testing)

ความสามารถในการทำงานร่วมกัน (Interoperability) ระหว่างการพัฒนาจากผู้ผลิตต่างรายเป็นสิ่งสำคัญอย่างยิ่งสำหรับโปรโตคอลมาตรฐาน เช่น TLS 1.3 ที่ต้องให้ผลิตภัณฑ์จากผู้ผลิตต่าง ๆ สื่อสารกันได้

**กิจกรรม Interoperability Testing สำหรับ PQC:**

- **IETF Hackathon:** กลุ่มทำงาน IETF จัดงาน Hackathon สม่ำเสมอเพื่อให้ผู้ผลิตทดสอบ Interoperability ของ PQC ใน TLS, SSH และโปรโตคอลอื่น ๆ
- **Open Quantum Safe Interop Testing:** โครงการ OQS มีสภาพแวดล้อมสำหรับทดสอบ Interoperability ระหว่าง Implementation ต่าง ๆ ของอัลกอริทึม NIST PQC
- **CloudFlare และ Google Interop Tests:** ผลการทดสอบ Interoperability แบบสาธารณะระหว่างการพัฒนา Chrome, CloudFlare และ OQS-OpenSSL ช่วยตรวจสอบว่ามาตรฐาน TLS Hybrid ใช้งานได้ในทางปฏิบัติ

## ห้องปฏิบัติการทดสอบที่ได้รับการรับรอง

### ATL ภายใต้ CMVP

ห้องปฏิบัติการทดสอบอิสระที่ได้รับการรับรองจาก NVLAP (National Voluntary Laboratory Accreditation Program) เพื่อทดสอบตาม FIPS 140-3 ได้แก่:

| ห้องปฏิบัติการ | ประเทศ | ความเชี่ยวชาญ |
| -- | -- | -- |
| **Leidos** | สหรัฐอเมริกา | FIPS 140-3, Hardware/Software Modules |
| **UL (Underwriters Laboratories)** | สหรัฐอเมริกา | FIPS 140-3, Common Criteria |
| **atsec information security** | สหรัฐอเมริกา, เยอรมนี | FIPS 140-3, Common Criteria, ITSEC |

สำหรับปี 2025-2026 ห้องปฏิบัติการเหล่านี้กำลังพัฒนาขีดความสามารถและเครื่องมือสำหรับทดสอบ ML-KEM, ML-DSA และ SLH-DSA โดยเฉพาะ เนื่องจากกระบวนการทดสอบ PQC แตกต่างจากอัลกอริทึมคลาสสิกอย่างมีนัยสำคัญ

### ห้องปฏิบัติการในเอเชีย

ปัจจุบันยังมีห้องปฏิบัติการ FIPS 140-3 ATL ที่ได้รับการรับรองจำนวนน้อยมากในภูมิภาคเอเชีย ซึ่งหมายความว่าองค์กรไทยที่ต้องการให้ผลิตภัณฑ์ของตนได้รับการรับรอง FIPS 140-3 อาจต้องส่งผลิตภัณฑ์ไปทดสอบในสหรัฐอเมริกาหรือยุโรป ซึ่งมีค่าใช้จ่ายและระยะเวลาเพิ่มขึ้น

ห้องปฏิบัติการในภูมิภาคที่เกี่ยวข้อง ได้แก่:
- **Korea Internet & Security Agency (KISA):** ทำหน้าที่ทดสอบและรับรองผลิตภัณฑ์รหัสลับในเกาหลีใต้ และมีความร่วมมือกับ NIST ในด้าน PQC
- **TeleCommunication Technology Association (TTA) ของเกาหลีใต้:** ออกใบรับรองสำหรับผลิตภัณฑ์โทรคมนาคมและความปลอดภัย
- **JNSA (Japan Network Security Association):** ให้แนวทางและกรอบการทดสอบสำหรับผลิตภัณฑ์รักษาความปลอดภัยในญี่ปุ่น

## สถานะปัจจุบันของการรับรอง PQC

### ผลิตภัณฑ์ที่ได้รับการรับรองในช่วงเปลี่ยนผ่าน

เนื่องจากมาตรฐาน FIPS สำหรับ PQC เพิ่งออกในเดือนสิงหาคม 2024 ยังไม่มีผลิตภัณฑ์ที่ได้รับการรับรอง FIPS 140-3 สำหรับ PQC โดยเฉพาะ ณ ต้นปี 2026 อย่างไรก็ตาม มีผู้ผลิตหลายรายที่ประกาศแผนการยื่นขอการรับรองหรือกำลังอยู่ในกระบวนการ:

| ผู้ผลิต | ผลิตภัณฑ์/โมดูล | สถานะ (ต้นปี 2026) |

### ผลิตภัณฑ์ที่ผ่านการทดสอบ CAVP

แม้ว่าการรับรองระดับโมดูล (FIPS 140-3) จะยังไม่พร้อม แต่การทดสอบ CAVP ผ่านระบบ ACVP ของ NIST ดำเนินการได้เร็วกว่า Implementation หลายรายการของ ML-KEM และ ML-DSA ที่อิงจาก liboqs และ Bouncy Castle ได้ผ่านการทดสอบ ACVP แล้ว ซึ่งถือเป็นหลักฐานเบื้องต้นของความถูกต้อง

## บทบาทของมาตรฐานไทยและการรับรองในประเทศ

ปัจจุบัน (ณ ต้นปี 2026) ประเทศไทยยังไม่มีกรอบการรับรองระดับชาติที่ครอบคลุม PQC โดยเฉพาะ อย่างไรก็ตาม มีหน่วยงานและกระบวนการที่เกี่ยวข้อง:

- **สำนักงานพัฒนาธุรกรรมทางอิเล็กทรอนิกส์ (ETDA):** กำหนดมาตรฐานและแนวปฏิบัติสำหรับลายเซ็นอิเล็กทรอนิกส์และ PKI ในประเทศไทย ซึ่งจะต้องปรับตัวรองรับ PQC ในอนาคต
- **สำนักงานคณะกรรมการกิจการกระจายเสียง กิจการโทรทัศน์ และกิจการโทรคมนาคมแห่งชาติ (กสทช.):** กำกับดูแลมาตรฐานความปลอดภัยสำหรับบริการโทรคมนาคม
- **ธนาคารแห่งประเทศไทย (ธปท.):** ออกแนวนโยบายด้านความปลอดภัยไซเบอร์สำหรับสถาบันการเงิน ซึ่งอ้างอิงมาตรฐานสากลรวมถึง FIPS 140-3

สำหรับการจัดซื้อผลิตภัณฑ์รหัสลับในภาคราชการไทย มักอ้างอิงมาตรฐาน Common Criteria (ISO/IEC 15408) และ FIPS 140-3 เป็นเกณฑ์ในการประเมิน ดังนั้นการติดตามสถานะการรับรองของผลิตภัณฑ์จึงมีความสำคัญสูง

## คำแนะนำเชิงปฏิบัติสำหรับองค์กรไทย

### ระยะสั้น (พ.ศ. 2568-2569)

1. **ตรวจสอบสถานะการรับรองก่อนจัดซื้อ:** ใช้ฐานข้อมูล CMVP ของ NIST และสอบถามผู้ผลิตโดยตรงเกี่ยวกับ Roadmap การรับรอง FIPS 140-3 สำหรับ PQC โดยเฉพาะ
2. **ยอมรับ Hybrid Implementation ชั่วคราว:** ผลิตภัณฑ์ที่ใช้ Hybrid KEM (เช่น X25519MLKEM768) ยังอยู่บนรากฐาน ECC ที่ได้รับการรับรองแล้ว และ NIST ยอมรับกลยุทธ์นี้เป็นแนวปฏิบัติที่ดีในช่วงเปลี่ยนผ่าน
3. **ทดสอบ Conformance ด้วยตนเองเบื้องต้น:** ใช้ Test Vectors จาก NIST ACVP เพื่อตรวจสอบ Implementation ที่ใช้งาน โดยเฉพาะสำหรับ Library ที่พัฒนาหรือปรับแต่งเอง

### ระยะกลาง (พ.ศ. 2569-2571)

4. **วางแผนงบประมาณสำหรับการรับรอง:** สำหรับผลิตภัณฑ์ที่ต้องการการรับรอง FIPS 140-3 ควรเริ่มวางแผนงบประมาณสำหรับกระบวนการทดสอบกับ ATL ล่วงหน้า ซึ่งมีค่าใช้จ่ายโดยประมาณตั้งแต่ \$50,000 ถึง \$200,000 ขึ้นอยู่กับความซับซ้อนของโมดูล
5. **ติดตาม Protection Profile สำหรับ PQC:** เมื่อ CCRA ออก Protection Profile สำหรับผลิตภัณฑ์ที่ใช้ PQC องค์กรในภาครัฐและภาคการเงินควรนำ PP เหล่านี้มาใช้เป็นเกณฑ์ในการจัดซื้อ
6. **สร้างความสัมพันธ์กับห้องปฏิบัติการทดสอบ:** ติดต่อห้องปฏิบัติการ ATL ล่วงหน้าเพื่อทำความเข้าใจกระบวนการและระยะเวลาในการทดสอบ เนื่องจาก Backlog ของการรับรองมักยาวนาน

### สิ่งที่ควรระวัง

- **อย่าสับสนระหว่าง "อ้างว่าสอดคล้อง" กับ "ได้รับการรับรองอย่างเป็นทางการ":** ผู้ผลิตหลายรายโฆษณาว่าผลิตภัณฑ์ "รองรับ PQC" หรือ "สอดคล้องกับ NIST" โดยไม่ได้ผ่านกระบวนการรับรองอย่างเป็นทางการ สำหรับระบบที่ต้องการความปลอดภัยสูง ควรเรียกหลักฐานการทดสอบที่เป็นรูปธรรม
- **ระวัง PQC ที่ยังไม่ได้มาตรฐาน:** อัลกอริทึมบางตัว เช่น NewHope หรือ NTRU ที่ไม่ได้รับการรับรองจาก NIST ควรหลีกเลี่ยงสำหรับระบบสำคัญ เนื่องจากอาจมีช่องโหว่ที่ยังไม่ถูกค้นพบ
- **ติดตามการอัปเดตมาตรฐานอย่างต่อเนื่อง:** ชุมชนรหัสลับยังคงวิเคราะห์ความปลอดภัยของอัลกอริทึม NIST PQC อยู่ หากมีการค้นพบช่องโหว่ใหม่ มาตรฐานอาจได้รับการปรับปรุง ซึ่งจะส่งผลต่อสถานะการรับรองของผลิตภัณฑ์ที่ใช้งานอยู่
