# เครื่องมือสำหรับเทคโนโลยี PQC ในปัจจุบัน

การเปลี่ยนผ่านสู่ยุคหลังควอนตัมไม่ใช่เพียงการเลือกอัลกอริทึมใหม่ แต่ยังต้องอาศัยเครื่องมือซอฟต์แวร์ที่หลากหลายเพื่อนำ PQC ไปใช้งานจริงในระบบสารสนเทศขององค์กร เครื่องมือเหล่านี้ครอบคลุมตั้งแต่ไลบรารีรหัสลับพื้นฐาน ไปจนถึงกรอบการทำงานสำหรับ TLS, PKI, การทดสอบ และการตรวจนับสินทรัพย์ทางรหัสลับ (Cryptographic Inventory) การทำความเข้าใจภูมิทัศน์ของเครื่องมือในปัจจุบันจะช่วยให้ทีมงานด้านความปลอดภัยไซเบอร์ขององค์กรไทยสามารถเลือกใช้ เครื่องมือที่เหมาะสมกับบริบท ข้อจำกัดทางเทคนิค และงบประมาณของตนได้อย่างมีประสิทธิภาพ

## กลุ่มของเครื่องมือ PQC

เครื่องมือในระบบนิเวศ PQC ปัจจุบันสามารถแบ่งออกเป็น 7 กลุ่มหลัก ซึ่งแต่ละกลุ่มมีบทบาทที่แตกต่างกันในวงจรชีวิตของการย้ายระบบ (Migration Lifecycle)

### ไลบรารีรหัสลับพื้นฐาน (Core Cryptographic Libraries)

ไลบรารีกลุ่มนี้เป็นรากฐานสำคัญที่สุดของระบบนิเวศ PQC โดยทำหน้าที่ implement อัลกอริทึมหลังควอนตัมในระดับต่ำสุด นักพัฒนาและผู้เชี่ยวชาญด้านความปลอดภัยส่วนใหญ่จะไม่ได้เรียกใช้งานไลบรารีเหล่านี้โดยตรง แต่จะได้รับประโยชน์จากผ่านชั้นซอฟต์แวร์ที่สูงกว่า เช่น OpenSSL

คุณลักษณะสำคัญของไลบรารีกลุ่มนี้:
- ให้ API สำหรับการสร้างกุญแจ (Key Generation), การห่อหุ้มกุญแจ (Encapsulation/Decapsulation) และการสร้าง/ตรวจสอบลายเซ็น (Sign/Verify)
- รองรับมาตรฐาน NIST ได้แก่ ML-KEM, ML-DSA, SLH-DSA และ FN-DSA
- บางไลบรารีมีการปรับปรุงประสิทธิภาพผ่านชุดคำสั่ง AVX2/AVX-512 เพื่อเพิ่มความเร็วบนฮาร์ดแวร์สมัยใหม่

### ไลบรารีโปรโตคอลเครือข่าย (Network Protocol Libraries)

ไลบรารีกลุ่มนี้ทำหน้าที่เชื่อมต่ออัลกอริทึม PQC เข้ากับโปรโตคอลความปลอดภัยที่ใช้งานจริง เช่น TLS 1.3, DTLS, SSH และ IPsec ซึ่งเป็นโปรโตคอลที่ถูกใช้ในทุกการสื่อสารบนอินเทอร์เน็ตปัจจุบัน การอัปเกรดไลบรารีเหล่านี้จึงมีผลต่อระบบจำนวนมากพร้อมกัน

ประเด็นที่ต้องพิจารณา:
- ความเข้ากันได้กับ TLS 1.3 Cipher Suites แบบ Hybrid (เช่น X25519MLKEM768)
- การรองรับ FIPS 140-3 ซึ่งกำลังอยู่ในกระบวนการรับรองสำหรับอัลกอริทึม PQC ใหม่
- ผลกระทบต่อขนาด TLS Handshake ที่เพิ่มขึ้นเนื่องจากกุญแจ PQC มีขนาดใหญ่กว่า

### ตัวห่อหุ้มภาษาโปรแกรม (Language Wrappers and SDKs)

ตัวห่อหุ้มภาษา (Language Bindings/Wrappers) ช่วยให้นักพัฒนาสามารถเรียกใช้งานฟังก์ชัน PQC ได้จากภาษาโปรแกรมระดับสูงโดยไม่ต้องเขียนโค้ด C โดยตรง ซึ่งช่วยลดความซับซ้อนและความเสี่ยงจากการเขียนโค้ดรหัสลับด้วยตนเอง

ภาษาที่มี Wrapper พร้อมใช้งาน ได้แก่ Python, Java, Go, Rust, C#, JavaScript (Node.js) และ Ruby โดยส่วนใหญ่อยู่ภายใต้โครงการ Open Quantum Safe (OQS)

### ระบบ PKI และการจัดการใบรับรอง (PKI and Certificate Management)

การเปลี่ยนผ่านสู่ PQC ต้องการการอัปเดตห่วงโซ่ใบรับรอง (Certificate Chain) และ PKI ทั้งระบบ เครื่องมือกลุ่มนี้ช่วยให้ CA (Certificate Authority) และระบบจัดการใบรับรองสามารถออก, ตรวจสอบ และจัดการใบรับรองที่ใช้อัลกอริทึมหลังควอนตัมหรือแบบ Hybrid ได้

ประเด็นสำคัญ:
- ใบรับรอง X.509 สำหรับ PQC มีขนาดใหญ่กว่าใบรับรอง RSA/ECC หลายเท่าตัว ซึ่งอาจกระทบต่อระบบที่มีข้อจำกัดด้านขนาดข้อมูล
- การรองรับ Hybrid Certificate ที่บรรจุทั้งกุญแจ ECC และ PQC ไว้ในใบรับรองเดียวเพื่อช่วงเปลี่ยนผ่าน

### เครื่องมือทดสอบและวัดประสิทธิภาพ (Testing and Benchmarking Tools)

ก่อนการนำ PQC ไปใช้งานในระบบจริง องค์กรต้องการเครื่องมือสำหรับทดสอบความถูกต้องของการ implement อัลกอริทึม (Correctness Testing) และวัดประสิทธิภาพบนฮาร์ดแวร์เป้าหมาย (Performance Benchmarking) โดยเฉพาะในสภาพแวดล้อมที่มีทรัพยากรจำกัด เช่น IoT

เครื่องมือกลุ่มนี้มักให้:
- Test Vectors อ้างอิงจาก NIST เพื่อตรวจสอบความถูกต้องของการ implement
- ผลการวัดประสิทธิภาพ (Cycle Counts) บนสถาปัตยกรรมต่าง ๆ เช่น x86-64, ARM Cortex-M4
- การวิเคราะห์ความปลอดภัยต่อการโจมตีผ่านช่องทางข้างเคียง (Side-Channel Analysis)

### เครื่องมือตรวจนับและค้นหารหัสลับ (Cryptographic Discovery and Inventory Tools)

ก่อนที่องค์กรจะสามารถย้ายระบบไปยัง PQC ได้ จะต้องรู้ก่อนว่ามีการใช้รหัสลับที่ใดบ้างในระบบสารสนเทศ เครื่องมือกลุ่มนี้ทำการสแกนโค้ด, ไฟล์การกำหนดค่า (Configuration Files), การรับส่งข้อมูลเครือข่าย และ Artifact ต่าง ๆ เพื่อสร้างบัญชีรายการการใช้งานรหัสลับ (Cryptographic Bill of Materials หรือ CBOM)

ความสำคัญในบริบทของไทย:
- ระบบสารสนเทศขนาดใหญ่ในองค์กรไทยมักมีการใช้งานรหัสลับกระจายอยู่ในหลายร้อยหลายพันจุด ซึ่งยากต่อการตรวจสอบด้วยมือ
- การสร้าง CBOM เป็นขั้นตอนแรกที่จำเป็นตามแนวทางของ NIST IR 8547 และ ETSI TS 103 744

### เครื่องมือและเฟิร์มแวร์สำหรับ HSM (HSM Firmware and Management Tools)                                                                                                                    
             
Hardware Security Module (HSM) เป็นอุปกรณ์ฮาร์ดแวร์เฉพาะทางที่ทำหน้าที่จัดเก็บและใช้งานกุญแจรหัสลับในสภาพแวดล้อมที่ปลอดภัยทางกายภาพ (Tamper-resistant Environment) HSM ถูกใช้อย่างแพร่หลายในองค์กรทางการเงิน, หน่วยงานภาครัฐ และโครงสร้างพื้นฐาน PKI เครื่องมือกลุ่มนี้ครอบคลุมทั้งเฟิร์มแวร์ที่อัปเดตให้รองรับ PQC และซอฟต์แวร์ Middleware สำหรับบริหารจัดการ HSM      

ประเด็นเฉพาะของ HSM ในบริบทPQC:                                                                                                                                                           
- HSM รุ่นปัจจุบันส่วนใหญ่ออกแบบมาเพื่อรองรับกุญแจ RSA และ ECC ขนาดเล็ก พื้นที่จัดเก็บที่ปลอดภัย (Secure Storage) และหน่วยประมวลผลภายในอาจไม่รองรับกุญแจ ML-KEM หรือ ML-DSA ที่มีขนาดใหญ่กว่าถึง 10-20 เท่า               
- มาตรฐาน PKCS#11 (Cryptoki) ซึ่งเป็น API มาตรฐานสำหรับการสื่อสารกับ HSM กำลังอยู่ในระหว่างการขยายเพื่อรองรับ PQC โดย OASIS ซึ่งหมายความว่าซอฟต์แวร์ที่ใช้ PKCS#11 API อยู่แล้วจะสามารถรองรับ PQC ได้เมื่อ HSM และ Driver พร้อม                                                                                                                                                                                 
- ผู้ผลิต HSM ชั้นนำ เช่น Thales (Luna HSM), Entrust (nShield) และ Utimaco กำลังทยอยออกเฟิร์มแวร์อัปเดตที่รองรับ ML-KEM และ ML-DSA สำหรับรุ่นฮาร์ดแวร์ใหม่ ส่วนรุ่นเก่าบางรุ่นอาจต้องเปลี่ยนฮาร์ดแวร์ใหม่ (Hardware Refresh)                                                                                                                                                                                      
- SoftHSM2 เป็น Open Source HSM จำลอง (Software Emulation) ที่ใช้สำหรับทดสอบระบบที่ต้องการ PKCS#11 Interface โดยไม่ต้องมีฮาร์ดแวร์จริง ปัจจุบันมีโครงการชุมชนที่กำลัง Integrate liboqs เข้ากับ SoftHSM2 เพื่อให้รองรั
    +บการทดสอบ PQC ผ่าน PKCS#11   

## โครงการ Open Source ที่เป็นที่นิยม

### liboqs — Open Quantum Safe

**ที่มา:** โครงการ Open Quantum Safe (OQS) ริเริ่มโดยมหาวิทยาลัย Waterloo ประเทศแคนาดา
**ภาษา:** C
**ใบอนุญาต:** MIT License

liboqs คือไลบรารีหลักของระบบนิเวศ Open Quantum Safe ที่ implement อัลกอริทึม PQC ทั้งหมดที่ได้รับการรับรองจาก NIST รวมถึงอัลกอริทึมที่อยู่ในระหว่างการพิจารณา โดยมีจุดเด่นสำคัญคือ Application Programming Interface (API) ที่เป็นมาตรฐานเดียวกันสำหรับอัลกอริทึมทุกตัว ทำให้สามารถเปลี่ยนอัลกอริทึมได้โดยแทบไม่ต้องแก้ไขโค้ดระดับบน

อัลกอริทึมที่รองรับ:
- **Key Encapsulation Mechanisms (KEM):** ML-KEM (Kyber), FrodoKEM, BIKE, HQC, Classic McEliece
- **Digital Signatures:** ML-DSA (Dilithium), SLH-DSA (SPHINCS+), FN-DSA (Falcon), MAYO, CROSS

liboqs ถูกใช้เป็นฐานของโปรเจกต์ OQS อื่น ๆ ทั้งหมด เช่น OQS-OpenSSL, OQS-BoringSSL และ Language Wrappers ต่าง ๆ ทำให้เป็นองค์ประกอบสำคัญที่สุดในระบบนิเวศ PQC แบบ Open Source

### OQS-Provider สำหรับ OpenSSL 3

**ที่มา:** โครงการ Open Quantum Safe
**ภาษา:** C
**ใบอนุญาต:** MIT License

OpenSSL 3.x นำเสนอสถาปัตยกรรม Provider ที่ช่วยให้สามารถเพิ่มความสามารถด้านรหัสลับใหม่ได้โดยไม่ต้องแก้ไข OpenSSL core OQS-Provider ใช้ประโยชน์จากสถาปัตยกรรมนี้เพื่อนำ PQC เข้าสู่ระบบนิเวศ OpenSSL ซึ่งเป็นไลบรารี TLS ที่ใช้งานแพร่หลายที่สุดในโลก

ประโยชน์สำหรับองค์กรไทย:
- ระบบที่ใช้ OpenSSL อยู่แล้วสามารถเพิ่มความสามารถ PQC ได้โดยการโหลด Provider ใหม่โดยไม่ต้องเปลี่ยนโค้ดแอปพลิเคชัน
- รองรับการทำ TLS Handshake แบบ Hybrid (เช่น X25519MLKEM768) ซึ่งเป็นแนวปฏิบัติที่แนะนำสำหรับช่วงเปลี่ยนผ่าน
- ใช้ทดสอบร่วมกับ OQS-Demos สำหรับการทดสอบ HTTPS และ SSH ด้วย PQC

### PQClean

**ที่มา:** ชุมชนวิจัย PQC นานาชาติ (ผู้มีส่วนร่วมจากหลายสถาบัน)
**ภาษา:** C, Assembly
**ใบอนุญาต:** Public Domain / MIT (แล้วแต่อัลกอริทึม)

PQClean มีจุดประสงค์เพื่อให้ "Implementation อ้างอิงที่สะอาด" (Clean Reference Implementations) ของอัลกอริทึม PQC ซึ่งผ่านการตรวจสอบความถูกต้องอย่างเข้มงวด โดยเน้นที่ความถูกต้อง (Correctness) และความสามารถในการอ่านและตรวจสอบโค้ด (Auditability) มากกว่าความเร็ว

PQClean กำหนดหลักเกณฑ์การ implement ที่ชัดเจน เช่น:
- ห้ามใช้ทรัพยากรแบบ Dynamic Memory Allocation ในบางกรณี
- ต้องเป็น Constant-time เพื่อป้องกัน Timing Side-Channel Attacks
- ต้องผ่านการทดสอบด้วย Test Vectors อย่างครบถ้วน

PQClean เป็นแหล่งโค้ดที่ liboqs และไลบรารีอื่น ๆ มักดึงมาใช้เป็นฐาน และเป็น Reference สำคัญสำหรับผู้ที่ต้องการ Audit หรือตรวจสอบความถูกต้องของอัลกอริทึม

### Bouncy Castle

**ที่มา:** Legion of the Bouncy Castle Inc. ประเทศออสเตรเลีย
**ภาษา:** Java, C#
**ใบอนุญาต:** MIT License

Bouncy Castle เป็นไลบรารีรหัสลับที่ครอบคลุมและเป็นที่นิยมสูงในระบบนิเวศ Java และ .NET โดยรองรับอัลกอริทึมรหัสลับคลาสสิกมานานกว่า 20 ปี ปัจจุบันได้เพิ่มการรองรับอัลกอริทึม PQC ที่ได้รับการรับรองจาก NIST ครบถ้วนแล้วในเวอร์ชัน 1.77 เป็นต้นมา

จุดเด่นสำหรับองค์กรที่พัฒนาระบบด้วย Java หรือ .NET:
- รองรับ ML-KEM, ML-DSA, SLH-DSA และ FN-DSA ทั้งใน Java และ C#
- มี API ระดับสูงสำหรับการสร้างและตรวจสอบใบรับรอง X.509 แบบ Hybrid
- ใช้งานในระบบ Enterprise จำนวนมากในไทย โดยเฉพาะในระบบธนาคารและ e-Government ที่สร้างด้วย Java

### AWS-LC (Amazon Web Services Libcrypto)

**ที่มา:** Amazon Web Services
**ภาษา:** C, Assembly
**ใบอนุญาต:** Apache 2.0 License (พร้อม OpenSSL Compatibility Layer)

AWS-LC คือ Fork ของ BoringSSL (ไลบรารีรหัสลับของ Google) ที่ Amazon ปรับปรุงให้มีประสิทธิภาพสูงและรองรับ PQC โดยสมบูรณ์ AWS ใช้ AWS-LC เป็นไลบรารีหลักสำหรับบริการต่าง ๆ เช่น AWS TLS, AWS KMS และ S3 ปัจจุบัน AWS-LC รองรับ ML-KEM และอยู่ระหว่างการรองรับ ML-DSA และ SLH-DSA

เหตุผลที่น่าสนใจสำหรับองค์กรที่ใช้ AWS:
- ได้รับการรับรอง FIPS 140-3 (อยู่ระหว่างกระบวนการ) ทำให้เหมาะกับองค์กรที่มีข้อกำหนดด้านการปฏิบัติตามกฎระเบียบ
- มีการปรับปรุงประสิทธิภาพระดับ Assembly สำหรับทั้ง x86-64 และ ARM64
- เป็น Drop-in Replacement สำหรับ OpenSSL ในหลายกรณี

**ภาษา:** C, Assembly (ARM Cortex-M4)
**ใบอนุญาต:** Public Domain / MIT

pqm4 (Post-Quantum Cryptography for Cortex-M4) เป็นโครงการที่มุ่งเน้นการ implement และ Benchmark อัลกอริทึม PQC บนไมโครคอนโทรลเลอร์ ARM Cortex-M4 ซึ่งเป็น Platform ที่ใช้กันอย่างแพร่หลายในอุปกรณ์ IoT, ระบบฝังตัว และอุปกรณ์อุตสาหกรรม

ความสำคัญของ pqm4:
- ให้ข้อมูลประสิทธิภาพ (Cycle Counts, Stack Usage, Code Size) ที่แม่นยำสำหรับแต่ละอัลกอริทึมบน Cortex-M4
- ช่วยให้ผู้ออกแบบระบบ IoT สามารถตัดสินใจได้ว่าอัลกอริทึมใดเหมาะสมกับข้อจำกัดของอุปกรณ์ตน
- มีการปรับปรุงโค้ดให้ใช้ RAM น้อยที่สุดผ่านเทคนิค Stack-sharing และการประมวลผลแบบแบ่งส่วน (Streaming)

### SoftHSM2

**ที่มา:** OpenDNSSEC Project (ปัจจุบันดูแลโดย SUNET และชุมชน Open Source)
**ภาษา:** C++
**ใบอนุญาต:** BSD 2-Clause License

SoftHSM2 เป็น Software Emulation ของ Hardware Security Module (HSM) ที่ implement PKCS#11 API (Cryptoki) มาตรฐานอย่างครบถ้วน ช่วยให้นักพัฒนาและผู้ดูแลระบบสามารถทดสอบแอปพลิเคชันที่ต้องการ HSM Interface โดยไม่จำเป็นต้องมีฮาร์ดแวร์จริง SoftHSM2 ถูกใช้งานอย่างแพร่หลายในการพัฒนาและทดสอบระบบ PKI, DNSSEC และแอปพลิเคชันที่ต้องการการจัดการกุญแจในสภาพแวดล้อมที่ปลอดภัย

ความสำคัญในบริบท PQC:
- ช่วยให้องค์กรสามารถทดสอบการผสานรวม PQC เข้ากับระบบที่ใช้ PKCS#11 ได้โดยไม่ต้องรอให้ HSM ฮาร์ดแวร์รองรับ PQC ซึ่งอาจใช้เวลาอีกหลายปี
- ชุมชน Open Source กำลังพัฒนาการผสานรวม liboqs เข้ากับ SoftHSM2 เพื่อให้รองรับการสร้าง, จัดเก็บ และใช้งานกุญแจ ML-KEM และ ML-DSA ผ่าน PKCS#11 Interface
- เหมาะสำหรับ Proof of Concept และการทดสอบขั้นตอนการ Integrate PQC เข้ากับ PKI หรือระบบจัดการใบรับรอง (Certificate Management System) ที่ใช้ PKCS#11 เช่น EJBCA หรือ HashiCorp Vault

ข้อควรระวัง:
- SoftHSM2 ไม่ได้ให้ความปลอดภัยทางกายภาพ (Physical Security) ใด ๆ กุญแจถูกจัดเก็บในรูปแบบไฟล์บนระบบปฏิบัติการ ดังนั้นจึงเหมาะสำหรับสภาพแวดล้อม Development และ Testing เท่านั้น ไม่ควรนำไปใช้ใน Production
- ในสภาพแวดล้อม Production ที่ต้องการ HSM จริง ควรตรวจสอบ Roadmap ของผู้ผลิต HSM ที่ใช้งานอยู่สำหรับการรองรับ PQC โดยตรง
