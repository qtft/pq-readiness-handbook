<!-- https://pqshield.com/wp-content/uploads/2021/02/PQShield-Quantum-Threat-2-The-First-NIST-Post-Quantum-Cryptographic-Standards-July-2022.pdf -->

การเลือกใช้อัลกอริทึม PQC ที่เหมาะสมสำหรับองค์กรไม่ใช่เพียงการเลือกอัลกอริทึมที่ได้รับการรับรองจาก NIST เท่านั้น แต่ต้องพิจารณาถึง **ระดับความมั่นคงปลอดภัย** (Security Level) และ **ประสิทธิภาพการประมวลผล** (Performance) ที่เหมาะสมกับบริบทของแต่ละระบบ ข้อมูลเชิงเปรียบเทียบในบทนี้จะช่วยให้ผู้บริหารและผู้เชี่ยวชาญด้านความปลอดภัยไซเบอร์ขององค์กรไทยสามารถตัดสินใจเลือกชุดพารามิเตอร์ที่สมดุลระหว่างความปลอดภัยและภาระทางเทคนิคได้อย่างมีหลักการ

## ระดับความมั่นคงปลอดภัยของ NIST (NIST Security Categories)

NIST กำหนดระดับความมั่นคงปลอดภัย (Security Category หรือ Security Level) ไว้ 5 ระดับ โดยใช้ความยากของการโจมตีที่ดีที่สุดในปัจจุบันเป็นเกณฑ์อ้างอิง ไม่ว่าจะเป็นการโจมตีด้วยคอมพิวเตอร์ทั่วไปหรือคอมพิวเตอร์ควอนตัม {cite}`nist2024fips203`

| ระดับ | เกณฑ์อ้างอิง | ความปลอดภัยเทียบเท่า |
| -- | -- | -- |
| **Level 1** | ยากไม่น้อยกว่าการค้นหา Key แบบ Brute-force ของ AES-128 | 128-bit classical / ~64-bit quantum (Grover's algorithm) |
| **Level 2** | ยากไม่น้อยกว่าการหา Collision ของ SHA-256 | เทียบเท่าความต้านทาน Collision ของ SHA-256 |
| **Level 3** | ยากไม่น้อยกว่าการค้นหา Key แบบ Brute-force ของ AES-192 | 192-bit classical / ~96-bit quantum |
| **Level 4** | ยากไม่น้อยกว่าการหา Collision ของ SHA-384 | เทียบเท่าความต้านทาน Collision ของ SHA-384 |
| **Level 5** | ยากไม่น้อยกว่าการค้นหา Key แบบ Brute-force ของ AES-256 | 256-bit classical / ~128-bit quantum |

**หมายเหตุสำคัญ:** ระดับความปลอดภัยเหล่านี้วัดเทียบกับ **ความยากของการโจมตีต่ออัลกอริทึม** ไม่ใช่ต่อระบบโดยรวม การรักษาความปลอดภัยของระบบยังขึ้นอยู่กับปัจจัยอื่น เช่น การจัดการกุญแจ การ implement และโปรโตคอลที่ใช้ร่วมกัน

### ชุดพารามิเตอร์ของแต่ละมาตรฐาน NIST PQC

**ML-KEM (FIPS 203) — Key Encapsulation Mechanism** {cite}`nist2024fips203`

| ชุดพารามิเตอร์ | ระดับความปลอดภัย | ลักษณะการใช้งานที่แนะนำ |
| -- | -- | -- |
| ML-KEM-512 | Level 1 | ระบบที่มีทรัพยากรจำกัด เช่น IoT, อุปกรณ์ฝังตัว |
| ML-KEM-768 | Level 3 | **ระดับมาตรฐานทั่วไป** เหมาะสำหรับการใช้งานส่วนใหญ่ |
| ML-KEM-1024 | Level 5 | ระบบที่ต้องการความปลอดภัยสูงสุด เช่น โครงสร้างพื้นฐานสำคัญ, PKI Root CA |

**ML-DSA (FIPS 204) — Digital Signature** {cite}`nist2024fips204`

| ชุดพารามิเตอร์ | ระดับความปลอดภัย | ลักษณะการใช้งานที่แนะนำ |
| -- | -- | -- |
| ML-DSA-44 | Level 2 | ลายเซ็นทั่วไปที่ต้องการขนาดเล็กกว่า ML-DSA-65 |
| ML-DSA-65 | Level 3 | **ระดับมาตรฐานทั่วไป** เหมาะกับใบรับรอง X.509 และ TLS |
| ML-DSA-87 | Level 5 | ลายเซ็นสำหรับเอกสารหรือระบบที่มีอายุใช้งานยาวนาน |

**SLH-DSA (FIPS 205) — Hash-Based Digital Signature** {cite}`nist2024fips205`

SLH-DSA มีชุดพารามิเตอร์ 12 ชุด แบ่งตามฟังก์ชัน Hash ที่ใช้ (SHA2 หรือ SHAKE), ระดับความปลอดภัย (128/192/256-bit) และลักษณะการปรับแต่ง (s = small signature / f = fast signing)

| ชุดพารามิเตอร์ | ระดับ | ลายเซ็น (Bytes) | ลักษณะ |
| -- | -- | -- | -- |
| SLH-DSA-SHA2-128s | Level 1 | 7,856 | ลายเซ็นเล็ก, สร้างช้า |
| SLH-DSA-SHA2-128f | Level 1 | 17,088 | ลายเซ็นใหญ่, สร้างเร็ว |
| SLH-DSA-SHA2-192s | Level 3 | 16,224 | ลายเซ็นเล็ก, สร้างช้า |
| SLH-DSA-SHA2-256s | Level 5 | 29,792 | ลายเซ็นเล็กที่สุดในระดับ 5 |
| SLH-DSA-SHA2-256f | Level 5 | 49,856 | ลายเซ็นใหญ่, สร้างเร็ว |

**FN-DSA (FIPS 206) — Falcon Digital Signature** {cite}`nist2024fips206`

| ชุดพารามิเตอร์ | ระดับความปลอดภัย | ลักษณะ |
| -- | -- | -- |
| Falcon-512 | Level 1 | ลายเซ็นเล็กมาก, แต่ Key Generation ช้าเนื่องจากใช้ Floating-point |
| Falcon-1024 | Level 5 | ลายเซ็นเล็กมากในระดับสูง, ต้องการฮาร์ดแวร์ที่รองรับ FP arithmetic |

### แนวทางการเลือกระดับความปลอดภัยสำหรับองค์กรไทย

สำหรับการใช้งานทั่วไปในองค์กร NIST และหน่วยงานด้านความปลอดภัยสากลแนะนำให้เริ่มต้นที่ **Level 3** (ML-KEM-768 และ ML-DSA-65) ซึ่งให้ความสมดุลระหว่างความปลอดภัย ขนาดของข้อมูล และประสิทธิภาพการประมวลผล

สำหรับข้อมูลที่มีอายุการใช้งานยาวนานหรือต้องปกป้องในระยะ 20-30 ปี ควรพิจารณา **Level 5** ซึ่งให้ขอบเขตความปลอดภัย (Security Margin) ที่สูงกว่า เพื่อรับมือกับความก้าวหน้าของเทคนิคการโจมตีที่อาจพัฒนาขึ้นในอนาคต

### ผลกระทบของขนาดข้อมูลต่อระบบต่าง ๆ

ขนาดของกุญแจและลายเซ็นที่ใหญ่ขึ้นส่งผลกระทบต่อระบบในหลายมิติ:

**1. ขนาดใบรับรอง X.509:**
ใบรับรอง TLS ปัจจุบันที่ใช้ ECDSA P-256 มีขนาดโดยทั่วไปประมาณ 1-2 KB เมื่อเปลี่ยนเป็น ML-DSA-65 ขนาดใบรับรองจะเพิ่มขึ้นเป็น 5-8 KB เนื่องจากกุญแจสาธารณะและลายเซ็นมีขนาดใหญ่กว่าอย่างมีนัยสำคัญ ซึ่งอาจทำให้ขนาดข้อมูล TLS Handshake รวมเกิน MTU ขนาด 1,500 Bytes

**2. ห่วงโซ่ใบรับรอง (Certificate Chain):**
ในระบบ PKI ทั่วไปที่มีใบรับรอง 3 ชั้น (Root CA → Intermediate CA → End-entity) ขนาดรวมของ Certificate Chain แบบ ML-DSA-65 อาจสูงถึง 20-25 KB เทียบกับ 4-6 KB ในแบบ ECDSA

**3. TLS Handshake Overhead:**
การศึกษาของ Cloudflare และ Google พบว่า TLS Handshake แบบ Hybrid (X25519MLKEM768) เพิ่มขนาดข้อมูลประมาณ 1.5-2 KB เมื่อเทียบกับ TLS แบบ ECDH ซึ่งยังอยู่ในระดับที่จัดการได้สำหรับเครือข่ายปัจจุบัน

## ต้นทุนการประมวลผล (Computational Costs)

### รอบการประมวลผลบนซอฟต์แวร์ (CPU Cycles)

ข้อมูลด้านล่างเป็นค่าประมาณการรอบ CPU (CPU Cycles) บนสถาปัตยกรรม x86-64 พร้อม AVX2 instruction set ซึ่งเป็นฮาร์ดแวร์ระดับ Server และ Desktop ทั่วไป อ้างอิงจากผลการ Benchmark ของโครงการ liboqs และ PQCBENCHMARKS {cite}`liboqs2024`

**อัลกอริทึม KEM (Key Encapsulation):**

| อัลกอริทึม | Key Generation | Encapsulation | Decapsulation |
| -- | -- | -- | -- |
| ECDH P-256 | ~70,000 | ~140,000 | ~140,000 |
| ML-KEM-512 | ~23,000 | ~28,000 | ~26,000 |
| **ML-KEM-768** | **~44,000** | **~50,000** | **~48,000** |
| ML-KEM-1024 | ~69,000 | ~80,000 | ~77,000 |

ML-KEM มีประสิทธิภาพดีกว่า ECDH อย่างน่าประหลาดใจในแง่รอบ CPU เนื่องจากอัลกอริทึมบน Module-Lattice สามารถปรับแต่งด้วย NTT (Number Theoretic Transform) และ AVX2 ได้อย่างมีประสิทธิภาพ

**อัลกอริทึม Digital Signature:**

| อัลกอริทึม | Key Generation | Signing | Verification |
| -- | -- | -- | -- |
| RSA-2048 | ~20,000,000 | ~70,000 | ~2,500 |
| ECDSA P-256 | ~70,000 | ~70,000 | ~130,000 |
| ML-DSA-44 | ~95,000 | ~180,000 | ~95,000 |
| **ML-DSA-65** | **~138,000** | **~240,000** | **~148,000** |
| ML-DSA-87 | ~206,000 | ~370,000 | ~204,000 |
| Falcon-512 (FN-DSA) | ~9,000,000 | ~220,000 | ~60,000 |
| Falcon-1024 (FN-DSA) | ~30,000,000 | ~450,000 | ~120,000 |
| SLH-DSA-SHA2-128s | ~37,000 | ~14,000,000 | ~350,000 |
| SLH-DSA-SHA2-128f | ~110,000 | ~2,600,000 | ~750,000 |
| SLH-DSA-SHA2-256s | ~95,000 | ~55,000,000 | ~1,400,000 |

**ข้อสังเกตสำคัญ:**

- **ML-DSA** มีเวลาในการ Sign เพิ่มขึ้นประมาณ 3-4 เท่าเมื่อเทียบกับ ECDSA แต่ยังอยู่ในระดับที่ Server สมัยใหม่รับได้สำหรับการใช้งานทั่วไป
- **Falcon (FN-DSA)** มีข้อเสียด้าน Key Generation ที่ช้ามากเนื่องจากต้องใช้ Floating-point arithmetic แต่เมื่อกุญแจถูกสร้างแล้ว การ Sign และ Verify รวดเร็วและลายเซ็นมีขนาดเล็กที่สุดในบรรดา PQC ทั้งหมด
- **SLH-DSA** มีจุดแข็งที่กุญแจสาธารณะขนาดเล็กมากและความปลอดภัยที่ไม่ขึ้นกับโครงสร้างทางคณิตศาสตร์ใด ๆ นอกจากฟังก์ชันแฮช แต่การ Sign ช้ามากโดยเฉพาะในชุดพารามิเตอร์แบบ -s

### การใช้หน่วยความจำ (Memory Usage)

การใช้ RAM เป็นประเด็นสำคัญโดยเฉพาะสำหรับระบบฝังตัวและ IoT:

| อัลกอริทึม | Stack Usage สำหรับ Signing (ประมาณ) | ความท้าทายหลัก |
| -- | -- | -- |
| ML-KEM-768 | ~4–6 KB | ต่ำ เหมาะกับระบบฝังตัว |
| ML-DSA-65 | ~60–70 KB | สูง ต้องการ RAM มากเพื่อประสิทธิภาพสูงสุด |
| Falcon-512 | ~25–30 KB | ปานกลาง ต้องการ FP unit |
| SLH-DSA-SHA2-128s | ~10–15 KB | ต่ำ แต่ Sign ช้ามาก |

ML-DSA ต้องการ Stack ขนาดใหญ่เพื่อเก็บ Intermediate Values ในกระบวนการ Rejection Sampling ซึ่งอาจเป็นปัญหาสำคัญบน Microcontroller ที่มี RAM เพียง 8-16 KB {cite}`kannwischer2019pqm4`

## สรุปเปรียบเทียบประสิทธิภาพ (Benchmark Summary)

### ซอฟต์แวร์บนสถาปัตยกรรมต่าง ๆ

ประสิทธิภาพของ PQC แตกต่างกันอย่างมีนัยสำคัญระหว่างสถาปัตยกรรมฮาร์ดแวร์:

**x86-64 พร้อม AVX2/AVX-512 (Server/Desktop ทั่วไป)**
- ML-KEM-768 และ ML-DSA-65 ทำงานได้ดีมาก เนื่องจาก NTT สามารถใช้ SIMD Instructions ได้อย่างเต็มประสิทธิภาพ
- ประสิทธิภาพสูงกว่า Scalar Implementation ประมาณ 3-5 เท่า
- เหมาะสำหรับ Web Server, API Gateway, และระบบ Authentication ที่ต้องรองรับการเชื่อมต่อจำนวนมาก

**ARM Cortex-A (Smartphone, SBC)**
- รองรับ NEON SIMD Instructions ช่วยเพิ่มประสิทธิภาพได้ประมาณ 2-3 เท่าเมื่อเทียบ Scalar
- ML-KEM-768 ทำงานได้ดีบน ARM Cortex-A53/A72 ระดับ ~100,000-200,000 รอบ
- เหมาะสำหรับ Mobile Application และ Edge Computing

**ARM Cortex-M4 (IoT, Microcontroller)** {cite}`kannwischer2019pqm4`
- ไม่มี FP Unit และ SIMD ที่สมบูรณ์ ทำให้ประสิทธิภาพลดลงอย่างมาก
- ML-KEM-512 ใช้เวลาประมาณ 900K-1.5M รอบ ยังพอใช้งานได้
- ML-DSA-44 ต้องการ RAM สูงถึง 68 KB สำหรับ Stack (เกินกว่าที่ Cortex-M4 ส่วนใหญ่มี)
- SLH-DSA แบบ Stateless เป็นตัวเลือกที่น่าสนใจสำหรับ IoT แม้ Sign จะช้า เพราะ Key Generation รวดเร็วและไม่ต้องการ RAM มาก

| -- | -- | -- | -- |
| **Scalar Software** | ARM Cortex-M4 ไม่มี SIMD | 1× (baseline) | ช้าที่สุด |

| **ASIC เฉพาะทาง** | Crypto Accelerator Chip | 100–500× | ประสิทธิภาพสูงสุด ต้นทุนสูง |

### ตัวอย่างการประเมินสำหรับระบบ TLS ที่มีการเชื่อมต่อจำนวนมาก

สมมติว่า Server ต้องรองรับ **10,000 TLS Handshakes ต่อวินาที** บน CPU ความเร็ว 3 GHz พร้อม AVX2:

**กรณีปัจจุบัน (ECDH P-256 + ECDSA P-256):**
- KeyGen + Sign + Verify รวมประมาณ 280,000 cycles
- ที่ 3 GHz = ~93 microseconds ต่อ Handshake
- ต้องการ CPU Cores ประมาณ 0.93 cores สำหรับ 10,000 req/s

**กรณี PQC Hybrid (X25519MLKEM768 + ML-DSA-65):**
- KeyGen (ECDH) + KeyGen (ML-KEM) + Sign (ML-DSA) + Verify (ML-DSA) ≈ 480,000 cycles
- ที่ 3 GHz = ~160 microseconds ต่อ Handshake
- ต้องการ CPU Cores ประมาณ 1.6 cores สำหรับ 10,000 req/s

ค่าประมาณนี้แสดงว่าการ Migrate ไปยัง PQC Hybrid **เพิ่มภาระ CPU ประมาณ 70%** สำหรับ TLS Handshake ซึ่งยังอยู่ในระดับที่ยอมรับได้สำหรับ Server ที่มีทรัพยากรเพียงพอ แต่ต้องวางแผนการ Capacity Planning ล่วงหน้า

## คำแนะนำสรุปสำหรับการเลือกพารามิเตอร์
