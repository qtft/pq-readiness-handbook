# ข้อจำกัดทางเทคโนโลยี
การเตรียมความพร้อมสู่ยุคควอนตัมขององค์กรในประเทศไทย ไม่ได้เป็นเพียงแค่เรื่องของการอัปเกรดซอฟต์แวร์ แต่ต้องผ่านการทำความเข้าใจในข้อจำกัดทางเทคโนโลยีที่ซับซ้อน ทั้งเทคโนโลยี PQC และ QKD เทคโนโลยีทั้งสองประเภทต่างมีขีดจำกัดที่องค์กรต้องพิจารณาอย่างรอบคอบ ทั้งในแง่ของประสิทธิภาพการคำนวณ ความต้องการฮาร์ดแวร์เฉพาะทาง ต้นทุนที่สูง และความยุ่งยากในการวางโครงสร้างพื้นฐานใหม่  การทำความเข้าใจข้อจำกัดเหล่านี้จะช่วยให้ผู้บริหารและผู้เชี่ยวชาญด้านความปลอดภัยไซเบอร์ของไทยสามารถประเมินความคุ้มค่าและเลือกใช้เทคโนโลยีที่เหมาะสมกับบริบทของแต่ละองค์กรได้ 

## ข้อจำกัดของเทคโนโลยี PQC
เทคโนโลยี PQC หรือรหัสลับหลังควอนตัม อาศัยปัญหาทางคณิตศาสตร์ที่เชื่อว่ามีความซับซ้อนเกินกว่าที่คอมพิวเตอร์ควอนตัมจะประมวลผลได้ในเวลาที่เป็นพหุนาม (Polynomial Time) เช่น ปัญหาบนแลตทิซ (Lattice-based Problems) | ปัญหาพหุนามหลายตัวแปร (Multivariate Cryptography) และปัญหาที่อ้างอิงตามแฮช (Hash-based Signatures)  มาตรฐานล่าสุดที่ได้รับการรับรองจากสถาบันมาตรฐานและเทคโนโลยีแห่งชาติ (NIST) เช่น ML-KEM (เดิมคือ Kyber) และ ML-DSA (เดิมคือ Dilithium) กลายเป็นกระดูกสันหลังใหม่ของการรักษาความปลอดภัยดิจิทัล แต่การนำไปใช้งานจริงกลับพบข้อจำกัดที่ส่งผลกระทบต่อระบบงานเดิมอย่างมีนัยสำคัญ

### ข้อจำกัดด้านประสิทธิภาพของซอฟต์แวร์
ข้อจำกัดด้านประสิทธิภาพของซอฟต์แวร์ PQC (Software Performance Limitation) เป็นผลสืบเนื่องโดยตรงมาจากขนาดของข้อมูลรหัสลับที่เพิ่มขึ้นอย่างมหาศาลเมื่อเทียบกับมาตรฐานคลาสสิก  การคำนวณบนโมดูลแลตทิซ (Module-Lattice) ใน ML-KEM และ ML-DSA ต้องจัดการกับพหุนามขนาดใหญ่และการสุ่มตัวอย่างค่าความผิดพลาด (Noise) ซึ่งสร้างภาระให้กับหน่วยประมวลผลกลาง (CPU) และหน่วยความจำ (RAM) ของระบบ

| คุณลักษณะ (Cryptographic Feature) | RSA-2048 | ECC (P-256) | ML-KEM-768 (L3) | ML-DSA-65 (L3) |
| -- | -- | -- | -- | -- |
| ขนาดกุญแจสาธารณะ (Public Key Size) | 256 Bytes |64 Bytes | 1,184 Bytes  | 1,952 Bytes  |
| ขนาดกุญแจส่วนตัว (Private Key Size) | 2,048 Bytes  | 32 Bytes | 2,400 Bytes  | 4,000 Bytes  |
| ขนาดข้อความรหัสลับ/ลายเซ็น (Ciphertext/Signature) | 256 Bytes | 64 Bytes | 1,088 Bytes  | 3,300 Bytes  |
| รอบการประมวลผล (Estimated CPU Cycles) |ปานกลาง-สูง |ต่ำ |ปานกลาง |สูง |

วามแตกต่างของขนาดข้อมูลนี้นำไปสู่ปัญหา "ภาระการสื่อสาร" (Communication Overhead) ที่รุนแรงขึ้น โดยเฉพาะในโปรโตคอลความปลอดภัยอย่าง TLS 1.3 หรือ IPsec ซึ่งเดิมออกแบบมาเพื่อรองรับกุญแจขนาดเล็กของ ECC เมื่อต้องส่งกุญแจสาธารณะและข้อความรหัสลับที่มีขนาดใหญ่กว่าเดิม 10 ถึง 20 เท่า ข้อมูลอาจเกินขนาดของ MTU (Maximum Transmission Unit) มาตรฐานที่ 1,500 ไบต์ ส่งผลให้เกิดการแตกเซกเมนต์ของแพ็กเกจ (TCP Segmentation) ซึ่งเพิ่มโอกาสในการเกิดความหน่วง (Latency) และความผิดพลาดในการส่งข้อมูลในเครือข่ายที่ไม่เสถียร ในเชิงการคำนวณ แม้ว่า ML-KEM จะมีความเร็วในการทำ Key Generation และ Encapsulation ที่น่าประทับใจเมื่อเทียบกับ RSA แต่ความซับซ้อนของอัลกอริทึมในการทำ Number Theoretic Transform (NTT) เพื่อเพิ่มประสิทธิภาพการคูณพหุนามยังคงเป็นภาระที่หนักหน่วงสำหรับซอฟต์แวร์ที่ไม่ได้ปรับแต่ง กระบวนการนี้ต้องการหน่วยความจำสำหรับการจัดเก็บค่าคงที่และพารามิเตอร์จำนวนมาก ซึ่งหากซอฟต์แวร์มีการจัดการหน่วยความจำที่ไม่ดี (Poor Memory Management) จะส่งผลให้เกิดปัญหา Stack Overflow ในระบบปฏิบัติการขนาดเล็ก หรือเพิ่มอัตราการใช้พลังงานในเซิร์ฟเวอร์ที่ต้องรองรับการเชื่อมต่อจำนวนมหาศาล (High-volume Authentication) 

นอกจากนี้ อัลกอริทึม PQC ยังมีข้อจำกัดด้าน "ความแปรปรวนของเวลาในการประมวลผล" (Timing Variation) ในซอฟต์แวร์บางรุ่นที่ไม่ได้ออกแบบมาเป็น Constant-time ซึ่งอาจเปิดช่องโหว่ให้เกิดการโจมตีผ่านช่องทางข้างเคียง (Side-Channel Attacks) ได้ง่ายกว่ารหัสลับคลาสสิก  การออกแบบซอฟต์แวร์ที่ปลอดภัยจึงต้องแลกมาด้วยความเร็วที่ลดลงและการเขียนโปรแกรมที่ซับซ้อนขึ้นอย่างมาก

### ข้อจำกัดด้านฮาร์ดแวร์
การเปลี่ยนผ่านสู่รหัสลับหลังควอนตัม (PQC) ไม่ได้ส่งผลกระทบเพียงแค่ระดับซอฟต์แวร์ แต่สร้างภาระให้กับโครงสร้างพื้นฐานฮาร์ดแวร์เดิมอย่างมีนัยสำคัญ เนื่องจากอัลกอริทึมมาตรฐานอย่าง ML-KEM และ ML-DSA ถูกออกแบบบนพื้นฐานของปัญหาแลตทิซ (Lattice-based) ที่ต้องการการประมวลผลทางคณิตศาสตร์แบบเวกเตอร์และพหุนามขนาดใหญ่ ซึ่งฮาร์ดแวร์ยุคเก่าไม่ได้ถูกออกแบบมาเพื่อรองรับ

#### ข้อจำกัดสำหรับคอมพิวเตอร์ Desktop และ Server
สำหรับอุปกรณ์ระดับประสิทธิภาพสูง แม้จะมีทรัพยากรมากกว่า IoT แต่ยังคงเผชิญกับข้อจำกัดในเชิงประสิทธิภาพสูงสุด (Throughput) และความเข้ากันได้ของสถาปัตยกรรม:

1. การพึ่งพาชุดคำสั่งเฉพาะทาง (Instruction Set Dependency): ประสิทธิภาพของ PQC บน Desktop และ Server ขึ้นอยู่กับการมีชุดคำสั่งเช่น AVX-512 (Advanced Vector Extensions) ของ Intel และ AMD ซึ่งช่วยให้ประมวลผลพหุนามขนาด 512 บิตได้ในรอบสัญญาณเดียว หากคอมพิวเตอร์ในองค์กรเป็นรุ่นเก่าที่ไม่รองรับ AVX-512 จะต้องใช้การประมวลผลแบบเดิม (Scalar) ซึ่งช้ากว่าและอาจเกิดปัญหา "Throttling" หรือความร้อนสะสมจากการเร่งการประมวลผลหนักในระยะเวลาสั้นๆ ส่งผลให้ความเร็วของเครื่องลดลง

2. ปัญหาคอขวดของอุปกรณ์เครือข่ายและ HSM:

    - Network Interface Cards (NICs): เซิร์ฟเวอร์ที่ต้องรองรับการเชื่อมต่อจำนวนมหาศาล (High-traffic) จะพบปัญหาเมื่อขนาดกุญแจและลายเซ็นของ PQC (เช่น ML-DSA-65 ที่มีขนาดลายเซ็นกว่า 3,300 ไบต์) เกินค่า MTU (Maximum Transmission Unit) มาตรฐานที่ 1,500 ไบต์ ส่งผลให้อุปกรณ์เครือข่ายระดับฮาร์ดแวร์ต้องทำ IP Fragmentation ซึ่งเพิ่มภาระการทำงานของ CPU เซิร์ฟเวอร์และอาจถูกบล็อกโดย Middleboxes หรือ Firewall รุ่นเก่าที่มองว่าแพ็กเก็ตขนาดใหญ่ผิดปกติเป็นภัยคุกคาม
    
    - Hardware Security Modules (HSMs): HSM ส่วนใหญ่ในปัจจุบันถูกออกแบบมาเพื่อเก็บกุญแจ RSA/ECC ขนาดเล็ก (ไม่กี่ร้อยไบต์) พื้นที่จัดเก็บภายใน (Secure Storage) และอัลกอริทึมที่ฝังในชิป (On-chip Logic) จึงไม่เพียงพอที่จะรองรับกุญแจ PQC ที่มีขนาดใหญ่กว่าเดิม 10-20 เท่า การอัปเกรดจึงอาจต้องถึงขั้นเปลี่ยนฮาร์ดแวร์ใหม่ (Hardware Refresh) เพื่อติดตั้งตัวเร่งความเร็วเฉพาะทางสำหรับ NTT (Number Theoretic Transform)

#### ข้อจำกัดสำหรับอุปกรณ์ IoT และระบบฝังตัว (Embedded Systems)
อุปกรณ์ IoT เป็นกลุ่มที่ได้รับผลกระทบรุนแรงที่สุดเนื่องจากข้อจำกัดด้านทรัพยากรที่เข้มงวด:

1. ข้อจำกัดด้านหน่วยความจำ (RAM/ROM):
    - ไมโครคอนโทรลเลอร์รุ่นยอดนิยมอย่าง ARM Cortex-M4 มักมี RAM เพียง 8-16 KB ในขณะที่การสร้างลายเซ็น ML-DSA-65 อาจต้องการ RAM สูงถึง 68 KB เพื่อประสิทธิภาพสูงสุด 
    - แม้จะมีการปรับปรุงซอฟต์แวร์ให้ใช้ RAM น้อยลง (เช่น เหลือ 8 KB) แต่จะส่งผลให้การประมวลผลช้าลงอย่างมหาศาลจนไม่สามารถใช้งานจริงได้ในระบบที่ต้องการการตอบสนองแบบ Real-time 
    
2. การสิ้นเปลืองพลังงาน (Power Consumption): อัลกอริทึม PQC ใช้การประมวลผลที่ซับซ้อนกว่า ECC หลายเท่า อุปกรณ์ IoT ที่ใช้แบตเตอรี่จะพบว่าอายุการใช้งานลดลงอย่างรวดเร็วจากการที่ CPU ต้องทำงานในสถานะพลังงานสูง (High-power state) นานขึ้นเพื่อคำนวณรหัสลับ

3. การขาดตัวเร่งความเร็วทางฮาร์ดแวร์ (Lack of Crypto-Accelerators): ชิป IoT ปัจจุบันมีตัวเร่งสำหรับ AES หรือ SHA-2 แต่ไม่มีวงจรสำหรับ Keccak (SHA-3) หรือการคูณพหุนามที่เป็นพื้นฐานของมาตรฐาน NIST ใหม่ ทำให้ต้องประมวลผลผ่านซอฟต์แวร์ทั้งหมด (Software-based implementation) ซึ่งมีความเสี่ยงต่อการถูกโจมตีผ่านช่องทางข้างเคียง (Side-Channel Attacks) เช่น การวัดพลังงานหรือความร้อน เนื่องจากไม่มีการป้องกันระดับฮาร์ดแวร์

4. ยุทธศาสตร์การปรับตัว (Trade-off): เพื่อแก้ปัญหาพื้นที่เก็บกุญแจจำกัด อุปกรณ์ IoT อาจต้องใช้เทคนิค "Seed-based key generation" คือการเก็บเพียงค่าเริ่มต้น (Seed) ขนาดเล็กแล้วค่อยคำนวณขยายกุญแจเต็มทุกครั้งที่ใช้งาน แม้จะประหยัดเนื้อที่ ROM แต่จะเพิ่มภาระให้กับ CPU และเวลาในการทำงาน (Latency) ทุกครั้งที่มีการรับส่งข้อมูล


## ข้อจำกัดของเทคโนโลยี QKD
Quantum Key Distribution หรือ QKD นำเสนอความมั่นคงปลอดภัยในระดับอุดมคติผ่านกฎของฟิสิกส์ควอนตัม เช่น ทฤษฎีการห้ามทำซ้ำ (No-cloning Theorem) และหลักความไม่แน่นอนของไฮเซนเบิร์ก (Heisenberg Uncertainty Principle) ซึ่งทำให้การลอบดักฟังข้อมูลใดๆ สามารถถูกตรวจพบได้ทันที  อย่างไรก็ตาม ในบริบทของการนำไปใช้งานจริงเชิงพาณิชย์ QKD กลับต้องเผชิญกับข้อจำกัดที่รุนแรงด้านการลงทุนและโครงสร้างพื้นฐานทางกายภาพ

### ข้อจำกัดด้านราคา
ข้อจำกัดด้านราคา (Cost Limitation) เป็นปัจจัยลบที่ชัดเจนที่สุดสำหรับองค์กรที่พิจารณาใช้ QKD  การลงทุนในเทคโนโลยีนี้ต้องการงบประมาณที่สูงกว่าการทำ PQC หลายเท่าตัว เนื่องจากต้องจัดซื้ออุปกรณ์ฮาร์ดแวร์เฉพาะทางที่มีความละเอียดอ่อนสูง 

โครงสร้างต้นทุนของการวางระบบ QKD สำหรับองค์กรไทยสามารถแบ่งออกเป็น 4 ส่วนหลัก:

1. ต้นทุนอุปกรณ์ (Capital Expenditure: CapEx): ราคาของอุปกรณ์ส่ง-รับสัญญาณควอนตัม (Alice & Bob) อยู่ในช่วง $100,000 ถึง $500,000 ต่อคู่ลิงก์ (Point-to-Point) ซึ่งขึ้นอยู่กับประสิทธิภาพและระยะทางการส่ง  หากต้องการขยายเครือข่ายเป็นวงกว้าง ราคาอุปกรณ์จะทวีคูณตามจำนวนโหนด 

2. ต้นทุนโครงสร้างพื้นฐานไฟเบอร์: QKD ต้องการสายไฟเบอร์ออปติกคุณภาพสูง หากองค์กรไม่มีสายไฟเบอร์เป็นของตนเอง การเช่า "Dark Fiber" จากผู้ให้บริการโทรคมนาคมจะมีค่าใช้จ่ายประมาณ $1,000 - $5,000 ต่อกิโลเมตรต่อปี  และหากต้องติดตั้งสายใหม่ ค่าใช้จ่ายอาจสูงถึง $5,000 - $60,000 ต่อไมล์ 

3. ค่าธรรมเนียมซอฟต์แวร์และการบริหารจัดการ: ระบบ QKD สมัยใหม่มักมาพร้อมกับค่าใบอนุญาตใช้งานซอฟต์แวร์ (Software License) และค่าบริการบำรุงรักษารายปี (Maintenance Contracts) ซึ่งอาจสูงถึง $50,000 ต่อโหนด เพื่อรับประกันความแม่นยำและการอัปเดตระบบ 

4. ต้นทุนโหนดที่ได้รับความเชื่อถือ (Trusted Nodes): เนื่องจากระยะทางจำกัด การส่งข้อมูลข้ามจังหวัดต้องการสถานีรีเลย์ที่ปลอดภัยทางกายภาพและมีเจ้าหน้าที่ดูแลตลอด 24 ชั่วโมง ซึ่งมีค่าใช้จ่ายประมาณ 36,000 ยูโรต่อสถานีต่อปี เพียงเพื่อดูแลความปลอดภัยของฮาร์ดแวร์

จากข้อมูลนี้ องค์กรขนาดกลางและขนาดย่อม (SMEs) ในไทยแทบจะไม่สามารถเข้าถึงเทคโนโลยี QKD ได้ในปัจจุบัน และความคุ้มค่าของการลงทุน (ROI) ยังคงเป็นคำถามใหญ่ เนื่องจากเทคโนโลยีนี้มุ่งเน้นไปที่การป้องกัน "ภัยในอนาคต" ที่ยังไม่เกิดขึ้นจริงอย่างเป็นรูปธรรม  องค์กรที่เหมาะสมสำหรับการลงทุนในระยะแรกจึงจำกัดอยู่ที่สถาบันการเงินขนาดใหญ่ หน่วยงานความมั่นคง และผู้ให้บริการโครงสร้างพื้นฐานระดับชาติเท่านั้น

### ข้อจำกัดด้านโครงสร้างพื้นฐาน
ข้อจำกัดด้านโครงสร้างพื้นฐาน (Infrastructure Limitation) เป็นอุปสรรคทางกายภาพที่แก้ไขได้ยากด้วยซอฟต์แวร์  ปัญหาพื้นฐานที่สุดคือ "การสูญเสียสัญญาณในใยแก้วนำแสง" (Fiber Attenuation) ซึ่งส่งผลให้ระยะทางการส่งกุญแจจำกัดอยู่ที่ประมาณ 80 ถึง 120 กิโลเมตรในเชิงพาณิชย์ และไม่เกิน 300 กิโลเมตรในเชิงห้องปฏิบัติการ

ประเด็นข้อจำกัดที่สำคัญด้านโครงสร้างพื้นฐาน ได้แก่:สัญญาณไม่สามารถขยายได้ (Non-amplifiability): ต่างจากสัญญาณอินเทอร์เน็ตคลาสสิกที่สามารถใช้ตู้แอมพลิฟายเออร์เพิ่มพลังงานแสงได้ 

1. สัญญาณควอนตัมไม่สามารถขยายได้เพราะจะทำให้สถานะควอนตัมพังทลาย  ทำให้การส่งสัญญาณทางไกล (เช่น กรุงเทพฯ - สงขลา) ต้องอาศัยการวาง "โหนดที่ได้รับความเชื่อถือ" (Trusted Nodes) ทุกๆ 100 กม. ซึ่งโหนดเหล่านี้กลายเป็นจุดเปราะบางใหม่เพราะกุญแจลับต้องถูกถอดและเข้ารหัสใหม่ที่โหนดนั้น ๆ

2. ความซับซ้อนของการใช้สายร่วม (Multiplexing Challenges): การส่งสัญญาณ QKD ไปในสายไฟเบอร์เดียวกับการสื่อสารปกติ (Data Traffic) ทำได้ยากมาก เนื่องจากแสงจากการส่งข้อมูลคลาสสิกมีความเข้มมากกว่าโฟตอนเดี่ยวของควอนตัมหลายพันล้านเท่า  แม้ปัจจุบันจะมีเทคนิคการแยกย่านความถี่ (WDM) โดยใช้ C-band สำหรับควอนตัม และ O-band สำหรับข้อมูล แต่ระบบนี้ยังต้องการการออกแบบที่แม่นยำและอุปกรณ์คัดกรองสัญญาณรบกวน (Filters) ที่มีราคาแพง 

3. ความต้องการสภาพแวดล้อมที่เสถียร: อุปกรณ์ QKD มีความไวสูงต่อการสั่นสะเทือนและการเปลี่ยนแปลงของอุณหภูมิในห้องเครื่อง  สำหรับประเทศไทยที่มีความชื้นและอุณหภูมิสูง การรักษาสภาพแวดล้อมให้คงที่เพื่อไม่ให้เกิดการเลื่อนของเฟสแสง (Phase Drift) เป็นภาระด้านพลังงานและการจัดการที่เพิ่มขึ้นอย่างมาก 

4. ข้อจำกัดของ QKD ผ่านพื้นที่ว่างและดาวเทียม: สำหรับลิงก์ข้ามเกาะหรือพื้นที่ห่างไกล การใช้ดาวเทียม QKD เป็นทางเลือกเดียวที่เป็นไปได้ แต่ก็มีข้อจำกัดเรื่องสภาพอากาศ (เมฆ หมอก ฝุ่น) ที่ขัดขวางการเดินทางของโฟตอน และระยะเวลาที่ดาวเทียมโคจรผ่านสถานีภาคพื้นดินซึ่งจำกัดการสร้างกุญแจให้ทำได้เพียงช่วงเวลาสั้นๆ ต่อวัน

ความท้าทายเหล่านี้ทำให้การสร้าง "Quantum Internet" ในไทยต้องใช้เวลาและงบประมาณมหาศาล และอาจต้องการความร่วมมือระดับประเทศในการสร้าง "National Backbone" ที่รวมเอาความสามารถของทั้งดาวเทียมและโครงสร้างพื้นฐานภาคพื้นดินเข้าด้วยกัน

## บทสรุปและคำแนะนำสำหรับองค์กรไทย
จากการวิเคราะห์ข้อจำกัดทางเทคโนโลยีทั้งหมด จะเห็นได้ว่าความท้าทายที่สำคัญที่สุดขององค์กรไทยไม่ใช่เพียงการเปลี่ยนอัลกอริทึมในเชิงซอฟต์แวร์ แต่คือการเผชิญหน้ากับ "ขีดจำกัดทางทรัพยากรของฮาร์ดแวร์" (Hardware Resource Ceiling) ซึ่งส่งผลกระทบแตกต่างกันไปตามประเภทของอุปกรณ์ ดังนี้:

1. สำหรับระบบ Desktop และ Server: องค์กรต้องเผชิญกับวงรอบการเปลี่ยนถ่ายฮาร์ดแวร์ (Hardware Refresh) ที่เร็วขึ้น เพื่อจัดหาเซิร์ฟเวอร์ที่รองรับชุดคำสั่งเวกเตอร์รุ่นใหม่ (เช่น AVX-512) และ HSM ที่มีพื้นที่จัดเก็บกุญแจขนาดใหญ่เพียงพอ นอกจากนี้ยังต้องเตรียมพร้อมรับมือกับปัญหาความหน่วงของเครือข่ายที่เกิดจากการแตกเซกเมนต์ของแพ็กเกจ PQC ขนาดใหญ่ ซึ่งอาจกระทบต่อประสิทธิภาพของแอปพลิเคชันที่ต้องการการตอบสนองแบบเรียลไทม์

2. สำหรับระบบ IoT และโครงสร้างพื้นฐานอุตสาหกรรม: ข้อจำกัดด้าน RAM และพลังงานแบตเตอรี่คืออุปสรรคที่แท้จริง อุปกรณ์ IoT จำนวนมากในไทยอาจไม่สามารถอัปเกรดเป็น PQC ได้เลย และจำเป็นต้องใช้ยุทธศาสตร์การปรับปรุงความปลอดภัยแบบแยกส่วน (Offloading Tasks) หรือการเปลี่ยนอุปกรณ์ใหม่ที่มีตัวเร่งความเร็วทางฮาร์ดแวร์สำหรับ PQC โดยเฉพาะ

3. สำหรับข้อมูลความมั่นคงสูง: การลงทุนใน QKD ยังคงเป็นทางเลือกสำหรับองค์กรระดับชาติที่มีงบประมาณเพียงพอและมีโครงสร้างพื้นฐานใยแก้วนำแสงเป็นของตนเอง เพื่อป้องกันภัยคุกคามประเภท "Harvest Now, Decrypt Later" ได้อย่างเด็ดขาด
