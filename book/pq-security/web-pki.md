# โครงสร้างพื้นฐานกุญแจสาธารณะสำหรับเว็บ [TODO]

โครงสร้างพื้นฐานกุญแจสาธารณะสำหรับเว็บ (Web Public Key Infrastructure: Web PKI) คือระบบที่ใช้ในการพิสูจน์ตัวตนและสร้างความเชื่อถือบนอินเทอร์เน็ต โดยอาศัยใบรับรองดิจิทัล (Digital Certificate) เชื่อมโยงกุญแจสาธารณะ (Public Key) เข้ากับตัวตนของเจ้าของ เช่น เว็บไซต์หรือองค์กร Web PKI เป็นรากฐานของโปรโตคอล HTTPS ที่ผู้ใช้งานอินเทอร์เน็ตทุกคนพึ่งพาในการสื่อสารที่ปลอดภัย

อย่างไรก็ตาม เมื่อคอมพิวเตอร์ควอนตัมที่มีขีดความสามารถเพียงพอถูกพัฒนาขึ้น ระบบรหัสลับที่ Web PKI พึ่งพาในปัจจุบัน เช่น RSA และ Elliptic Curve Cryptography (ECC) จะไม่สามารถให้ความมั่นคงปลอดภัยได้อีกต่อไป การเปลี่ยนผ่าน Web PKI ทั้งระบบให้รองรับอัลกอริทึมยุคหลังควอนตัม (Post-Quantum Cryptography: PQC) จึงเป็นความท้าทายเชิงเทคนิคที่ต้องได้รับการแก้ไขในระดับโครงสร้างพื้นฐานอินเทอร์เน็ต

## องค์ประกอบของ Web PKI

- **ใบรับรอง (Certificate)** — ใบรับรองดิจิทัลมาตรฐาน X.509 ผูกกุญแจสาธารณะเข้ากับตัวตนของเจ้าของ โดยมี CA ลงนามรับรองความถูกต้อง เมื่อเบราว์เซอร์เชื่อมต่อผ่าน HTTPS เซิร์ฟเวอร์จะส่งห่วงโซ่ใบรับรอง (Certificate Chain) ให้เบราว์เซอร์ตรวจสอบทีละชั้นจนถึง Root CA ที่เชื่อถือ

- **ผู้ให้บริการออกใบรับรอง (Certification Authority: CA)** — CA มีโครงสร้างแบบลำดับชั้น โดย Root CA คือ trust anchor ที่ฝังอยู่ในระบบปฏิบัติการและเบราว์เซอร์ ปัจจุบันมีประมาณ 150 องค์กรทั่วโลก Root CA ออกใบรับรองให้กับ Intermediate CA (หรือ Subordinate CA) เท่านั้น ซึ่ง Intermediate CA ทำหน้าที่ออกใบรับรองให้ผู้ใช้งานปลายทาง การแบ่งชั้นนี้ช่วยจำกัดผลกระทบหาก Intermediate CA ถูกโจมตี

- **บันทึกความโปร่งใสของใบรับรอง (Certificate Transparency Log)** — CT Log คือ Append-Only Merkle Tree สาธารณะที่บันทึกใบรับรองทุกใบที่ CA ออก เบราว์เซอร์กำหนดให้ใบรับรองต้องมี Signed Certificate Timestamp (SCT) เป็นหลักฐานการบันทึก ช่วยให้เจ้าของโดเมนตรวจจับการออกใบรับรองปลอมได้

## ความท้าทายของ Web PKI ในยุคควอนตัม

การเปลี่ยนผ่าน Web PKI ไปสู่ PQC เผชิญกับความท้าทายสำคัญที่เกิดจากขนาดของลายมือชื่อดิจิทัลและกุญแจสาธารณะที่ใหญ่กว่าอัลกอริทึมแบบดั้งเดิมอย่างมีนัยสำคัญ ดังแสดงในตารางเปรียบเทียบ

## Merkle Tree Certificates

Merkle Tree Certificates (MTC) เป็นข้อเสนอสถาปัตยกรรม Web PKI รูปแบบใหม่ที่ออกแบบมาเพื่อแก้ปัญหาขนาดลายมือชื่อ PQC โดยใช้โครงสร้างข้อมูล Merkle Tree ในการรับรองใบรับรองจำนวนมากด้วยลายมือชื่อเดียว แทนที่การลงนามใบรับรองแต่ละใบแยกกันดังเช่นระบบ X.509 แบบดั้งเดิม

**หลักการทำงาน**

CA จะรวบรวมใบรับรองที่ออกในแต่ละช่วงเวลา (Batch) และสร้าง Merkle Tree โดยใบรับรองแต่ละใบเป็น Leaf Node ของต้นไม้ จากนั้น CA ลงนามเพียงครั้งเดียวที่ค่า Root Hash ของต้นไม้ด้วยลายมือชื่อ PQC หนึ่งลายมือชื่อ เพื่อรับรองใบรับรองทั้งหมดในชุดนั้นพร้อมกัน

เมื่อเบราว์เซอร์ต้องการตรวจสอบใบรับรองของเว็บไซต์ เซิร์ฟเวอร์จะส่ง Merkle Inclusion Proof ซึ่งเป็นหลักฐานทางคณิตศาสตร์ที่พิสูจน์ว่าใบรับรองนั้นอยู่ใน Merkle Tree ที่ CA ได้ลงนามรับรองแล้ว Inclusion Proof มีขนาดเพียง O(log N) โดย N คือจำนวนใบรับรองในชุด ทำให้ขนาดข้อมูลที่ต้องส่งในแต่ละ TLS Handshake ลดลงอย่างมีนัยสำคัญ

**ข้อดีหลัก**

- **ลดขนาดข้อมูลต่อการเชื่อมต่อ**: ลายมือชื่อ PQC หนึ่งลายมือชื่อรับรองใบรับรองหลายล้านใบ แทนที่การลงนามแต่ละใบ ทำให้ overhead ต่อการเชื่อมต่อลดลงอย่างมาก
- **ความโปร่งใสในตัว (Built-in Transparency)**: โครงสร้าง Merkle Tree มีคุณสมบัติความโปร่งใสในตัว จึงรองรับหน้าที่ของ Certificate Transparency Log ได้โดยธรรมชาติ
- **ความเข้ากันได้กับ PQC**: ออกแบบมาให้รองรับลายมือชื่อ PQC ที่มีขนาดใหญ่ตั้งแต่แรก

**ข้อจำกัดและความท้าทาย**

- **ความเข้ากันได้ย้อนหลัง (Backward Compatibility)**: MTC ต้องการการเปลี่ยนแปลงโครงสร้างพื้นฐานของ Web PKI ทั้งฝั่ง CA, เบราว์เซอร์ และเซิร์ฟเวอร์ จึงต้องมีช่วงเปลี่ยนผ่านที่รันระบบ X.509 เดิมควบคู่กันไป
- **ความล่าช้าในการออกใบรับรอง (Issuance Latency)**: CA ต้องรอรวบรวมใบรับรองเป็นชุดก่อนจึงจะสร้าง Merkle Tree และลงนามได้ ทำให้ใบรับรองอาจไม่พร้อมใช้งานทันทีหลังการออก
- **การเพิกถอน (Revocation)**: กลไกการเพิกถอนใบรับรองในระบบ MTC ยังอยู่ระหว่างการออกแบบและมีความซับซ้อนกว่าระบบ X.509 แบบดั้งเดิม

### มาตรฐาน IETF

IETF กำลังดำเนินการพัฒนา Draft Standard สำหรับ Merkle Tree Certificates สำหรับ TLS โดยข้อเสนอนี้มุ่งให้ Merkle Tree Certificates กลายเป็นกลไกมาตรฐานสำหรับ Web PKI ในยุคหลังควอนตัม  

แนวทาง MTC สะท้อนให้เห็นว่าการเปลี่ยนผ่านสู่ PQC ไม่ได้หมายถึงเพียงการเปลี่ยนอัลกอริทึมเท่านั้น แต่อาจต้องออกแบบสถาปัตยกรรมโครงสร้างพื้นฐานความมั่นคงปลอดภัยขึ้นใหม่เพื่อให้รองรับข้อจำกัดของอัลกอริทึม PQC ได้อย่างมีประสิทธิภาพ


The Internet Engineering Task Force (IETF) recently created a working group, [PKI, Logs, And Tree Signatures (“PLANTS”)](https://datatracker.ietf.org/wg/plants/about/)

[Draft Standard: Merkle Tree Certificates](https://datatracker.ietf.org/doc/draft-ietf-plants-merkle-tree-certs/)
